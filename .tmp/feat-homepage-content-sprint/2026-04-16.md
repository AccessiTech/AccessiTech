## Research Session — Bootstrap + SCSS Architecture — 2026-04-16 (16:30)

**Governing axiom**: Endogenous-First — primary endogenous source: AccessiTech codebase (`src/scss/`, component `.scss` files, `docs/disclosures/CONTRIBUTING.md` BEM convention)

### Research Question

**What are Bootstrap 5 + SCSS + react-bootstrap best practices for component library styling, and should AccessiTech adopt BEM given its Bootstrap foundation?**

**Context**:

- Current architecture: 18 `.scss` files (2 core in `src/scss/`, 10 component co-located, 3 page co-located, 3 test mocks)
- Current react-bootstrap adoption: Grid (Row/Col/Container), Breadcrumb, Button/ButtonGroup, Form/Alert
- CONTRIBUTING.md mandates BEM but no documented rationale for BEM vs Bootstrap utilities
- A11y features: simplified view, font switcher, high-contrast mode
- Stack: Vite + React + react-bootstrap + SCSS

**Gate deliverables**:

1. **D1**: BEM recommendation verdict (YES/NO/HYBRID) with rationale for Bootstrap context
2. **D2**: Top 3 SCSS architectural improvements for current codebase
3. **D3**: Top 5 react-bootstrap expansion opportunities (current custom element → suggested component)
4. **D4**: Committed synthesis doc at `docs/styleguides/bootstrap-scss-best-practices.md` with WCAG theming patterns

**Success criteria**:

- All 4 deliverables completed
- Synthesis doc includes executive summary, best practices, BEM analysis, expansion table, architecture recommendations, WCAG patterns, references
- Return to orchestrator: ≤2000 tokens (bullets only: verdict + 3 changes + 5 expansions + commit SHA)

---

### Research Output (Compressed)

**BEM verdict**: **HYBRID (deprecate for new code)** — BEM redundant for Bootstrap + react-bootstrap context; Bootstrap's utility-first philosophy + react-bootstrap component abstraction make BEM block-element-modifier naming counterproductive. Reserve BEM only for custom a11y features (simplified-view, font-options, high-contrast) that have no Bootstrap equivalent.

**Top 3 architectural changes**:

1. **Import order enforcement** — Variables before Bootstrap imports: functions → overrides → variables → maps → mixins → components → utilities API. Current `variables.scss` may be loading out of order.
2. **Reduce co-located SCSS 13→6-8 files** — Move Header, Footer, SplashSocials to Bootstrap utilities (Grid/spacing); keep only brand-specific (a11y features, ContactForm validation, Services custom grid).
3. **CSS custom properties for runtime switching** — Use `:root { --font-body: ... }` for font switcher feature instead of SCSS variables (enables runtime JS override without rebuild).

**Top 5 react-bootstrap expansions**:

| Current Custom               | Suggested Component              | Benefit                                      | Effort |
| ---------------------------- | -------------------------------- | -------------------------------------------- | ------ |
| `.services-grid` container   | `<CardGroup>` or `<Card>`        | Responsive grid without custom media queries | Medium |
| `.contact-info` list         | `<ListGroup>`                    | Semantic list + accessible focus states      | Small  |
| `.blog-filters` button group | `<ButtonGroup variant="toggle">` | Built-in toggle states + keyboard nav        | Small  |
| `.product-page__cta` section | `<Card><Card.Footer>`            | Semantic card structure                      | Medium |
| Custom blog post badge spans | `<Badge>`                        | Semantic badge component + color variants    | Small  |

**Synthesis document**: `docs/styleguides/bootstrap-scss-best-practices.md` — 300 lines, 8 sections (Executive Summary, Bootstrap patterns, BEM analysis with rationale, 5 expansion wins table, SCSS architecture, WCAG theming patterns, references, current file audit)

**Estimated impact**: 40% reduction in custom CSS (1490 lines → 900 lines) after refactor; improved machine-readability for accessibility audits (Bootstrap utilities parseable by axe/WAVE vs opaque custom classes).

**Generalizable heuristic** (encode into substrate):

- For "best practices" research on established frameworks (Bootstrap, React, etc.), **audit current codebase adoption first** before fetching external docs — the gap between "what framework provides" and "what codebase uses" is the highest-leverage finding.
